🔑 The Q/K/V Mechanism

Self-attention works by transforming each token into three roles: Query (what am I looking for?), Key (what do I offer?), Value (what info do I provide?).

💡 Analogy: Imagine searching for a job. Your Query is "what kind of role do I want?". Each job's Key describes what it offers. The match between your query and the job's key determines whether you'll click on it. The Value is the actual job description you take away if matched.

Step 1 — Project each token into Q, K, V
Linear projections create three views of each token

Each token embedding (e.g., 768-dim vector) is multiplied by three different learned weight matrices W_Q, W_K, W_V to produce three different vectors.

Q = X · W_Q K = X · W_K V = X · W_V
Query (Q)
What am I looking for?
Key (K)
What do I offer?
Value (V)
What info do I provide?
Step 2 — Compute attention scores
Dot product Q · Kᵀ tells how well each query matches each key

Higher dot product = better match = more relevant. Then divide by √d_k to prevent gradients from vanishing with large vectors.

scores = (Q · KT) / √d_k
Step 3 — Normalize with softmax
Convert raw scores into a probability distribution

Softmax makes each row sum to 1.0 — these are interpretable attention weights (what fraction of "attention" goes to each token).

weights = softmax(scores)
Step 4 — Weighted sum of Values
Combine the Values, weighted by attention

The output for each token is a context-enriched representation — a blend of all Values, mostly weighted by what the token most attended to.

Attention(Q,K,V) = softmax(QKT/√d_k) · V

🧠 Multi-Head Attention — Why Multiple Heads?

Instead of one attention operation, models run 8-128 parallel heads. Each head specializes in detecting different relationships — like having different "lenses" on the same sentence.

Head 1 — Syntax

Tracks subject-verb agreement and grammatical structure
"The cats are sleeping" → 'are' attends to 'cats' (plural agreement)

Head 2 — Coreference

Resolves pronouns and references like "it", "he", "they"
"The dog ate its food" → 'its' attends to 'dog'

Head 3 — Semantics

Identifies topic-relevant words and concepts
"Python is fast" → 'fast' attends to 'Python' (programming context)

Head 4 — Positional

Attends to nearby tokens (local context)
'word_t' attends strongly to 'word_t-1' and 'word_t+1'

Head 5 — Punctuation

Tracks clause boundaries and sentence structure
Words after a comma attend back to clause subject

Head 6 — Entity Type

Detects nouns vs verbs vs adjectives
"runs fast" → 'fast' attends to 'runs' (adverb pattern)

Head 7 — Discourse

Tracks topic flow and conversation context
In a chat, 'that' may attend to topics 2-3 sentences back

Head 8+ — Specialized

Many heads learn surprising or hard-to-name patterns
Some attend only to special tokens like [CLS] or [SEP]

Researchers don't program these specializations — they emerge naturally during training. You can discover what each head does by analyzing its attention patterns, but you cannot dictate it in advance. This is one of the most fascinating aspects of how Transformers learn.

🔀 Encoder vs Decoder vs Encoder-Decoder

Three architectures share the same Transformer building blocks but use them differently for different goals.

📖 Encoder-Only

Bidirectional self-attention — reads in both directions

  • Every token attends to EVERY other token (no masking)
  • Optimized for understanding, not generation
  • Output: rich context-aware embeddings
  • Best for: classification, NER, semantic search
Examples: BERT, RoBERTa, DistilBERT, sentence-transformers

✍️ Decoder-Only

Causal self-attention — can only see past tokens

  • Each token only attends to tokens BEFORE it (masked)
  • Optimized for generation — predict next token
  • Output: probability distribution over next token
  • Best for: chat, writing, code, reasoning
Examples: GPT-5, Claude 4.x, Llama 4, Mistral, Gemini (most modern LLMs)

🔄 Encoder-Decoder

Both — encoder reads, decoder generates with cross-attention

  • Encoder reads source input bidirectionally
  • Decoder generates output, attending to encoder + own past
  • Optimized for sequence-to-sequence transformation
  • Best for: translation, summarization, structured outputs
Examples: T5, BART, mT5, Whisper (audio-to-text)

Why decoder-only won the 2020s: Decoder-only models are simpler to scale, and the autoregressive "predict the next token" objective produces a model that naturally handles instruction-following, in-context learning, and generation. By 2022 it became clear that scale + decoder-only + RLHF was the winning recipe.
