Structured Outputs

Get reliable JSON from LLMs using three techniques — from prompt instructions to Pydantic schemas

Three Approaches to Structured Output

Choose the right technique based on your reliability requirements and production constraints.

Approach 1
Prompt Instructions
Tell the model to return JSON in your prompt. The simplest approach but relies on model compliance.
✓ No extra SDK features needed
✓ Works with any model
✓ Easy to implement
✗ May produce invalid JSON
✗ No schema enforcement
✗ Requires defensive parsing
Approach 2
JSON Mode
Pass response_format={"type":"json_object"} to guarantee valid JSON output (but no schema enforcement).
✓ Guarantees valid JSON
✓ Simple to enable
✓ Works with gpt-4o-mini
✗ Keys/types not enforced
✗ Still need validation
✗ Must mention JSON in prompt
Recommended
Approach 3
Structured Outputs (Pydantic)
Use .parse() with a Pydantic model. API guarantees output matches your schema exactly.
✓ Schema is fully enforced
✓ Types validated automatically
✓ Returns typed Python object
✗ Requires pydantic library
✗ Limited to supported models

Defensive JSON Parsing

Even with JSON mode, models occasionally produce malformed output. Use this multi-stage parsing approach as a fallback.

1
Direct parse
Try json.loads(raw) first. Succeeds ~95% of the time when JSON mode is enabled.
2
Strip markdown fences
Regex for ```json ... ``` blocks. Models sometimes wrap JSON in code fences even when instructed not to.
3
Find first JSON object
Regex for the first {...} block in the text. Handles models that prepend explanation text before the JSON.
4
Retry with correction prompt
Append "Your previous response was not valid JSON. Return ONLY the JSON object." and retry the API call. Maximum 3 retries.
Best practice: For production code, always use Approach 3 (Pydantic + .parse()). The API guarantees schema compliance so you don't need defensive parsing at all. Use defensive parsing only when working with models that don't support structured outputs.
