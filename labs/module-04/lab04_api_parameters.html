API Parameter Reference

Key parameters for the Chat Completions API. Understanding these lets you optimize quality, cost, and latency.

Parameter Type Default Description & When to Change
temperature float 1.0 Sampling randomness. 0=deterministic, 2=max creative.
Set 0.0 for extraction/classification. 0.7 for chat. Ignored on GPT-5 / o-series reasoning models — use reasoning_effort instead.
max_completion_tokens int ∞ Maximum tokens in the response (renamed from max_tokens). On reasoning models, this caps reasoning + visible output combined.
On GPT-5/o-series, budget for reasoning tokens too — a "high" effort task may eat 1000s before any visible output appears. Equivalent: max_output_tokens on the Responses API.
reasoning_effort str "medium" GPT-5 / o-series only. How much hidden reasoning the model does before answering: "minimal" · "low" · "medium" · "high".
"minimal" for classification / simple lookups. "high" for math proofs, agent planning. Each step up roughly 2–5× the latency and reasoning-token cost.
verbosity str "medium" GPT-5 only. Controls the length of the visible answer: "low" · "medium" · "high". Independent of reasoning_effort.
"low" for terse production output. Pair with reasoning_effort="high" when you want deep thinking but short answers.
top_p float 1.0 Nucleus sampling — limits token selection to the smallest set whose cumulative probability exceeds top_p. Alternative to temperature.
Don't change both temperature AND top_p at once. Also ignored on reasoning models.
frequency_penalty float 0.0 Penalizes tokens that appear frequently in the response so far (-2.0 to 2.0). Reduces repetition of the same words.
Use 0.1–0.5 when responses repeat the same phrases. Not supported on reasoning models.
presence_penalty float 0.0 Penalizes tokens that have appeared at all in the response (-2.0 to 2.0). Encourages introducing new topics.
Use 0.1–0.5 to encourage more diverse, wide-ranging responses. Not supported on reasoning models.
stop str | list null Up to 4 stop sequences. Generation halts when any is produced. The stop string itself is NOT included in the response.
Use ["###", "\n\n"] to stop at section breaks. Good for few-shot prompts.
seed int null Best-effort determinism. Same seed + same inputs → higher probability of same output. Not 100% guaranteed.
Use for reproducible experiments and regression testing your prompts.
stream bool false Stream tokens as they're generated. Reduces perceived latency for users. Same total cost as non-streaming.
Enable for chat UIs. Chat Completions yields token deltas (chunk.choices[0].delta.content); Responses yields typed events (response.output_text.delta, etc).
previous_response_id str null Responses API only. Pass the .id from the prior response to chain conversation turns server-side.
Use for multi-turn assistants on GPT-5/o-series so reasoning items are preserved across turns.
store bool true Responses API only. Whether OpenAI stores the request/response on the server (30-day retention).
Set store=False for data-residency-sensitive flows. You'll then need to resend history yourself.

Cost reduction tips:
• Default to a mini or nano tier — escalate only if quality is insufficient.
• On GPT-5/o-series, drop reasoning_effort to "low" or "minimal" for simple tasks — slashes hidden reasoning-token cost.
• Set max_completion_tokens to limit output length.
• Prompt caching is automatic — identical input prefixes get ~50–90% off on cached input tokens. Put stable instructions first.
• Use the Batch API (/v1/batches) for non-interactive jobs: 50% off, 24-hour SLA.

Streaming vs Non-Streaming

Streaming delivers tokens as they're generated, reducing perceived latency in user-facing applications. Same total cost, very different UX.

Non-Streaming
Waiting for complete response...
User waits 2–5s with no feedback. The full response appears all at once. Bad UX for chat interfaces.
Streaming
Waiting to start...
Tokens appear as generated. First token arrives in ~300ms. Feels much faster even though total time is the same.
Streaming — Chat Completions:
stream = client.chat.completions.create(model="gpt-4.1-mini", messages=[...], stream=True)
for chunk in stream:
  text = chunk.choices[0].delta.content or ""
  print(text, end="", flush=True)

Streaming — Responses API (typed events):
with client.responses.stream(model="gpt-4.1-mini", input="...") as stream:
  for event in stream:
    if event.type == "response.output_text.delta":
      print(event.delta, end="", flush=True)
  final = stream.get_final_response()
