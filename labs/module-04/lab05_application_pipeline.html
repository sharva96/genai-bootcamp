API Call with Retry Logic ▾ click to expand
Handle transient failures with exponential backoff. Rate limit errors (429) and server errors (5xx) are often temporary.
import time from openai import OpenAI, RateLimitError, APIStatusError client = OpenAI() def call_with_retry(messages: list, max_retries: int = 3) -> str: for attempt in range(max_retries): try: response = client.chat.completions.create( model="gpt-4o-mini", messages=messages, temperature=0.0, max_tokens=1024 ) return response.choices[0].message.content except RateLimitError: wait = 2 ** attempt # 1s, 2s, 4s print(f"Rate limited. Waiting {wait}s...") time.sleep(wait) except APIStatusError as e: if e.status_code >= 500: print(f"Server error {e.status_code}. Retrying...") time.sleep(2 ** attempt) else: raise # 4xx errors are not retryable raise RuntimeError(f"Failed after {max_retries} attempts")

Error Handling Patterns

The OpenAI Python SDK provides specific exception types. Catching the right exception lets you handle each error appropriately.

What it means

You're sending too many requests per minute (RPM) or too many tokens per minute (TPM). Each tier has limits. Free tier: 3 RPM, 200K TPM. Paid tier: 500+ RPM, 30M+ TPM.

Fix: Exponential backoff — wait before retrying. The API returns a Retry-After header with the suggested wait time.

Fix: Exponential Backoff
import time from openai import RateLimitError for attempt in range(5): try: return make_api_call() except RateLimitError as e: wait = 2 ** attempt # 1,2,4,8,16s print(f"Rate limit. Waiting {wait}s") time.sleep(wait) raise RuntimeError("Max retries exceeded")
What it means

Your API key is invalid, expired, or revoked. This is NOT retryable — retrying will always fail until you fix the key.

Fix: Check the OPENAI_API_KEY environment variable. Verify the key is active at platform.openai.com.

Handle: Fail Fast
from openai import AuthenticationError try: make_api_call() except AuthenticationError as e: # Do NOT retry — key is invalid logging.critical("Invalid API key. Check OPENAI_API_KEY.") raise SystemExit(1) # Hard stop
What it means

Network failure — can't reach the API. Possible causes: no internet, DNS failure, firewall, OpenAI outage. Often transient.

Fix: Retry with backoff. Set a timeout on the client to avoid hanging indefinitely.

Fix: Timeout + Retry
from openai import OpenAI, APIConnectionError # Set a 30-second timeout on the client client = OpenAI(timeout=30.0) try: response = client.chat.completions.create(...) except APIConnectionError as e: # Network error — safe to retry logging.warning(f"Connection error: {e}") raise # Let retry decorator handle it
Production-Ready Exception Handler
from openai import ( OpenAI, RateLimitError, AuthenticationError, APIConnectionError, APIStatusError, APITimeoutError ) def safe_call(messages, retries=3) -> str: client = OpenAI(timeout=30.0) for attempt in range(retries): try: resp = client.chat.completions.create( model="gpt-4o-mini", messages=messages ) return resp.choices[0].message.content except AuthenticationError: raise # never retry auth errors except (RateLimitError, APIConnectionError, APITimeoutError) as e: if attempt == retries - 1: raise time.sleep(2 ** attempt) except APIStatusError as e: if e.status_code >= 500: if attempt == retries - 1: raise time.sleep(2 ** attempt) else: raise # 4xx: don't retry

Common Application Architectures

Choose the right pattern for your use case. Most production applications combine multiple patterns.

⛓️ Sequential Pipeline
Chain multiple API calls, where each step's output feeds the next. Good for multi-stage processing like extract → classify → summarize.
Use when: Task is too complex for one prompt, quality benefits from decomposition, or each stage needs different models.
🔀 Parallel Fan-out
Call multiple models or prompts in parallel using asyncio. Aggregate results (voting, union, best-of). Trades cost for latency reduction.
Use when: Independent subtasks, self-consistency voting, or running the same prompt on multiple inputs simultaneously.
🛡️ Guard + Process
Run a cheap, fast classification call first to validate input safety or type, then route to the appropriate processing prompt.
Use when: User-facing apps need content moderation, or inputs require routing to different specialized prompts.
🔁 Generate + Validate
Generate output with one call, then validate it with a second call (or schema check). Retry if validation fails. High quality at 2× cost.
Use when: Output quality is critical, structured output format must be guaranteed, or self-correction is needed.

from pydantic import BaseModel from openai import OpenAI, RateLimitError from typing import Literal import time, logging client = OpenAI(timeout=30.0) class TicketResult(BaseModel): category: Literal["billing", "technical", "security", "general"] urgency: Literal["low", "medium", "high"] one_line_summary: str requires_human: bool def classify_ticket(ticket: str, retries: int = 3) -> TicketResult: if not ticket.strip(): raise ValueError("Ticket text is empty") for attempt in range(retries): try: resp = client.beta.chat.completions.parse( model="gpt-4o-mini", messages=[ {"role":"system", "content": "Classify support tickets accurately."}, {"role":"user", "content": ticket} ], response_format=TicketResult, temperature=0.0 ) logging.info(f"tokens={resp.usage.total_tokens}") return resp.choices[0].message.parsed except RateLimitError: if attempt == retries - 1: raise time.sleep(2 ** attempt) # Usage: result = classify_ticket("Can't login since 8am") print(result.urgency) # "high" print(result.category) # "technical"
What this example demonstrates: Secure client init · Pydantic schema with Literal constraints · .parse() for guaranteed schema compliance · Retry loop with exponential backoff · Token usage logging · Input validation before API call
