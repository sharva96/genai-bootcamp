Chat Completions & Responses API

Master message roles, multi-turn conversations, and the two ways OpenAI handles state — stateless (Chat Completions) and stateful (Responses)

The Three Message Roles

Every message in the Chat Completions API must have a role. The role tells the model who is "speaking" and how to interpret that message.

🧭
system
Sets the model's persona, behavioral rules, and constraints for the entire conversation. Processed first, before any user messages. Persists across all turns.
{
  "role": "system",
  "content": "You are a concise technical assistant. Answer in under 3 sentences."
}
👤
user
Input from the human — questions, instructions, or content to process. This is the primary "turn" in the conversation. Alternates with assistant messages.
{
  "role": "user",
  "content": "Explain what an API is."
}
🤖
assistant
The model's previous responses. Include these when building multi-turn conversations so the model knows what it said before. Without them, each call is independent.
{
  "role": "assistant",
  "content": "An API is an interface that allows software components to communicate."
}
Key insight (Chat Completions): The Chat Completions API has no memory between calls. To maintain conversation history, you must include all previous messages (user + assistant turns) in each request. This means context grows with every turn — and so does the cost.

Alternative (Responses API): The newer Responses API can store turns server-side. Pass previous_response_id=<prior> and OpenAI threads the conversation for you — no need to resend history. See the Stateful Chaining section below.

Stateful Chaining — the Responses API Alternative

Same three-turn dialog, two very different code paths. Toggle to compare.

# Chat Completions — you maintain history client-side from openai import OpenAI client = OpenAI() history = [{"role": "system", "content": "You are a Python tutor."}] def chat(user_msg: str) -> str: history.append({"role": "user", "content": user_msg}) response = client.chat.completions.create( model="gpt-4.1-mini", messages=history, # resend EVERY prior message every call ) reply = response.choices[0].message.content history.append({"role": "assistant", "content": reply}) return reply print(chat("What is a list comprehension?")) print(chat("Show me an example with squares.")) print(chat("How does that differ from a generator?")) # Cost: turn 3 sends turns 1+2+3 in the payload — grows linearly
# Responses API — OpenAI stores history server-side from openai import OpenAI client = OpenAI() last_id: str | None = None def chat(user_msg: str) -> str: global last_id response = client.responses.create( model="gpt-4.1-mini", instructions="You are a Python tutor.", input=user_msg, # just the NEW message — no history! previous_response_id=last_id, # chain to the previous turn ) last_id = response.id # remember for the next call return response.output_text print(chat("What is a list comprehension?")) print(chat("Show me an example with squares.")) print(chat("How does that differ from a generator?")) # Cost: each call sends only the new user message — payload stays small # Bonus: reasoning items from GPT-5/o-series are preserved across turns
Chat Completions wins when
You need full client-side control of history (compress, redact, branch). You're targeting non-OpenAI providers that follow the chat-completions shape. You want a simple stateless model where any call can be retried in isolation.
Responses wins when
You're building an agent or multi-turn assistant. You use GPT-5 / o-series and need reasoning items preserved across turns. You want built-in tools (web_search, file_search, code_interpreter) without writing executors.
Hybrid is fine
You can use both in the same app. Some teams use Responses for end-user chat and Chat Completions for batch/back-office workflows. Migrate gradually — the APIs are not wire-compatible but they're not exclusive either.
Storage caveat
Responses persistence defaults to store=True with ~30-day retention on OpenAI's servers. Pass store=False for data-residency-sensitive flows — but then you must resend history yourself, just like Chat Completions.

Context Window Management

As conversations grow, token costs and latency increase. Use these strategies to manage context window usage in production.

Sliding Window
Keep only the last N turns. Drop the oldest user+assistant pairs when the total token count exceeds a threshold. Simple and effective.
Summarization
Periodically summarize older conversation history into a single condensed message. Trade detail for token efficiency. Good for very long sessions.
Message Trimming
Remove or truncate individual messages that exceed a per-message token budget. Use tiktoken to count tokens before building your messages array.
Stateless per Turn
For task-based apps (not chat), don't maintain history at all. Each API call is independent. Use RAG to inject relevant context instead of raw history.
Responses Chaining
Switch to the Responses API and pass previous_response_id. OpenAI stores history server-side; per-call payload stays small. Best for agents and reasoning models.
Prompt Caching
Long system prompt or instructions? OpenAI auto-caches identical input prefixes — repeated calls get ~50–90% input-token discount. No code change; works on both APIs.
Counting tokens: Use the tiktoken library (uv add tiktoken) to count tokens before making API calls.
import tiktoken; enc = tiktoken.encoding_for_model("gpt-4.1-mini"); tokens = len(enc.encode(text))

Newer models (GPT-5, o-series) use the o200k_base encoding — fall back to tiktoken.get_encoding("o200k_base") if the model name isn't recognized.
