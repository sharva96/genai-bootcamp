Anatomy of a Chatbot

The seven layers that turn an LLM into a production chatbot

The Seven Layers

Click each layer to expand. Every production chatbot — no matter how simple it looks — decomposes into these seven layers.

1
Frontend / Channel
Web widget · Slack · CLI · Voice · SMS
▸

What it does

Where the user types. Determines latency expectations, UI affordances (streaming vs. all-at-once, markdown vs. plain text), and what non-text signals you have access to (button clicks, file uploads, voice).

Web · streaming + markdown
Slack · threads + 4k char limit
Voice · short utterances, no UI
CLI · power users, no rendering

Design rule

The same backend may need to fan out to multiple channels with different policies per channel. Keep the channel adapter thin — protocol translation only. All logic lives below.

2
Session & State Store
Conversation history · user profile · long-term memory
▸

What it does

The conversation is a sequence of (user, assistant) pairs scoped to a session_id. Without this layer the bot is amnesic — the LLM itself has no built-in memory.

  • In-memory dict — toy demos only; lost on restart.
  • Redis — fast, scoped to a session TTL.
  • Postgres / Firestore — durable, auditable, supports analytics.
  • Vector store — long-term per-user recall across sessions.

Design rule

Keep session state (this conversation) separate from user state (preferences, prior sessions). They have different retention and access patterns.

3
Input Guardrails
Moderation · PII filter · scope check · rate limit
▸

What it does

Cheap checks before you spend tokens on a request you do not want to serve.

if moderation.check(user_msg).flagged:
    return safe_decline()

if pii_detector.has_secrets(user_msg):
    user_msg = redact(user_msg)

if rate_limiter.exceeded(user_id):
    return too_many_requests()
Moderation API ≈ free
Saves ~$0.01 per blocked turn
4
Context Builder
system prompt + retrieval + memory + new turn
▸

What it does

The most application-specific layer. Assembles the messages array by combining the persistent system prompt, retrieved documents (RAG), a windowed slice of history, optional tool definitions, and the new user message.

messages = [
    {"role": "system", "content": SYSTEM_PROMPT},
    {"role": "system", "content": format_retrieved(passages)},
    *sliding_window(history, n=6),
    {"role": "user", "content": user_msg},
]

Design rule

Critical instructions and the most recent turns belong at the edges of the prompt — the model attends to them more reliably. Avoid burying rules in the middle.

5
LLM Call
model + params + tools + streaming
▸

What it does

The actual model invocation. Pick a model family (GPT-5, GPT-4.1, GPT-4o-mini), set parameters (temperature, max_tokens), wire up tools, and stream or buffer.

gpt-4o-mini — chat default
gpt-4o — quality
gpt-5-mini — reasoning
stream=True — TTFT < 800ms

Cost rule

Multi-turn chat is dominated by input cost (history re-sent each turn). Reducing history is the highest-leverage cost optimization. max_tokens caps worst-case output spend.

6
Output Guardrails
moderation · format check · hallucination filter
▸

What it does

Post-generation checks. Did the model output something unsafe, off-format, or in violation of policy? Did it leak the system prompt or hallucinate?

  • Run moderation on the output — same API as input.
  • Validate structured outputs against schema; retry on parse failure.
  • Strip system-prompt content if it leaks.
  • Fact-check claims against retrieved sources.
7
Telemetry
logs · traces · evals · user feedback
▸

What it does

Log every turn — inputs, outputs, tokens, latency, tool calls, errors. Without telemetry you cannot diagnose regressions, run evaluations, or improve the bot.

session_id · turn_id
prompt + response (redacted)
tokens · cost · latency
👍 / 👎 user feedback

Design rule

Treat conversation logs as a product asset, not just debug output. Tomorrow's eval set and fine-tuning data live in today's logs.
