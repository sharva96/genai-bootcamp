Zero-Shot vs Few-Shot Explorer

Understand when and why to use examples in your prompts — and how to craft them effectively

Core Concepts

Two fundamental prompting strategies — each suited to different situations.

⚡
Zero-Shot Prompting

Ask the model to perform a task with no examples — relying entirely on the model's pre-trained world knowledge. The model must infer the expected behavior from the task description alone.

When it works well
Common tasks (translation, summarization) Exploratory use Fast iteration Simple classification
Limitations
Domain-specific formats Edge cases Novel task types
📋
Few-Shot Prompting

Provide the model with 2–10 examples of the desired input→output behavior before the actual task. The model pattern-matches the structure, format, and style of your demonstrations.

When it works well
Precise output formats (JSON keys) Domain-specific labels Edge case handling Consistent style
Limitations
Uses more tokens Wrong examples teach wrong behavior Diminishing returns after ~8 examples
How few-shot works (mechanically): The model doesn't "learn" from examples — no weights are updated. Instead, in-context examples shift the probability distribution over output tokens. The model pattern-matches the structure and style of your examples to generate the next output.

Guidelines for Effective Few-Shot Examples

The quality of your examples matters more than the quantity.

🎯
Cover Diverse Cases
Don't show only easy or obvious examples. Include edge cases, ambiguous inputs, and the hardest categories. The model will pattern-match to what you provide.
📐
Consistent Format
Every example must use the exact same input/output structure. If one example uses JSON and another uses plain text, the model will produce inconsistent outputs.
✅
Label Quality Matters
Incorrect labels in examples teach incorrect behavior. If your "positive" example is actually mixed sentiment, the model learns the wrong boundary. Review every example carefully.
🔢
3–5 Examples Usually Sufficient
More examples consume context window and API tokens with diminishing returns. Start with 3 covering your most important cases, then add more only if quality is still poor.
⏱️
Recency Bias Is Real
The most recent examples (closest to the actual input) have the most influence on the output. Put your most representative example last, just before the actual input.
📊
Match Your Real Distribution
Examples should reflect the actual distribution of inputs you'll encounter in production. If 70% of your real inputs are negative reviews, don't build examples with only positive ones.
