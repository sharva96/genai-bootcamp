Prompt Evaluation Dashboard

Six dimensions to evaluate before declaring a prompt production-ready. A prompt that passes 3 test cases is not necessarily ready to ship.

Accuracy
94%
"Is the output factually correct or correctly classified?"
Measure: % correct on a labeled test set of 50+ diverse inputs. Must include hard cases.
Format Compliance
98%
"Does output match the required structure?"
Measure: % of responses that are valid JSON / correct table format / correct list structure.
Consistency
82%
"Does the same input always produce equivalent output?"
Measure: Run same input 5x with temperature=0. Track variance in classification or key fields.
Robustness
76%
"Does it handle edge cases and adversarial inputs?"
Measure: Pass rate on adversarial inputs: empty strings, very long text, special chars, other languages.
Latency
P95: 2.1s
"How fast does it respond at scale?"
Measure: P50 and P95 response time across 100+ real calls. Track separately per model.
Token Efficiency
420 avg
"How many tokens consumed per call?"
Measure: Average tokens (prompt + completion) per request × expected monthly volume = projected cost.
Deployment threshold recommendation: Accuracy ≥ 90%, Format compliance ≥ 95%, Consistency ≥ 90% on a test set of 50+ inputs. Robustness and efficiency are secondary but should be tracked from day one.

Optimization Technique Reference

Six key techniques — click any row to expand details.

Temperature Tuning
Most impactful for: structured outputs, classification consistency, creative tasks that need variability
Free
Use 0.0 for classification and extraction. Use 0.3–0.5 for summarization. Use 0.7–1.0 for brainstorming. Never use >0.3 for JSON output — inconsistency rises sharply. Check that temperature is set per-call, not just default-assumed.
Prompt Decomposition
Use when a single prompt requires more than 2–3 sequential operations (extract, classify, summarize, act)
+Latency
Break a complex task into a pipeline of 2–5 targeted prompts. Each prompt does one thing well. Pass the output of one prompt as the input to the next. This improves accuracy on each step and makes debugging much easier — you can see exactly where the pipeline breaks.
Chain-of-Thought (CoT)
Use for reasoning-heavy tasks: math, logic, multi-step analysis, eligibility decisions
+Tokens
Add "Think step by step" or "Work through this step by step, then give your final answer." The model "writes out" intermediate reasoning, which becomes context for the final answer. CoT dramatically improves accuracy on arithmetic, logical, and multi-step decision problems. The reasoning tokens add cost but reduce errors significantly.
Self-Consistency
Use for high-stakes reasoning where accuracy is critical and cost/latency tolerance is high
High cost (Nx)
Generate 5–7 responses to the same question with temperature=0.7, then take the majority vote. This averages out individual model errors. Best for: exam-style reasoning, critical decisions, cases where a single model error is costly. Tradeoff: 5× the cost and latency. Use sparingly in production — only where the accuracy gain justifies the cost.
Negative Examples
Use when zero-shot or standard few-shot produces a specific, reproducible error on a category
+Tokens
Show the model what BAD output looks like, labeled as incorrect, alongside the correct version. This is especially effective for: classification boundary errors (model confuses two adjacent categories), over-eager classifications (model marks everything as urgent), systematic format errors. Add: "BAD EXAMPLE — do not do this: [wrong output]" followed by "GOOD EXAMPLE: [correct output]".
Context Window Optimization
Use when prompt length is high, costs are a concern, or latency is sensitive to prompt size
Free
Techniques: (1) Abbreviate labels: "pos/neg/neu" saves 2 tokens per occurrence. (2) Use compact JSON keys: {"s":"pos","c":0.9} vs {"sentiment":"positive","confidence":0.9}. (3) Strip document boilerplate before inserting. (4) Move static instructions to the system prompt (cached). (5) Truncate documents to relevant sections using a fast/cheap model first. Token costs are usage × price/token — optimization compounds over millions of calls.
