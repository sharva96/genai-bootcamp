Output Structuring Techniques

Three approaches to reliable structured output — from basic to production-grade.

Prompt Code
prompt = """ Extract structured information from this biography. Return ONLY valid JSON. Do not include any explanation or markdown code fences. Required fields: - name (string) - age (integer or null) - occupation (string) - skills (array of strings) - location (string or null) Biography: {bio} """ response = client.chat.completions.create( model="gpt-4o-mini", messages=[{"role": "user", "content": prompt}], temperature=0.0 # deterministic )
✓ Works with any model, no special API features needed
✓ No additional dependencies
✗ Model may occasionally wrap JSON in markdown code blocks
✗ No guarantee of valid JSON — need defensive parsing
Expected Output
{ "name": "Maria Chen", "age": 34, "occupation": "Machine Learning Engineer", "skills": [ "PyTorch", "distributed training", "MLOps", "LLM fine-tuning" ], "location": "San Francisco" }
Defensive Parsing Tip: Always wrap json.loads() in a try/except. Strip markdown fences if present. Log and retry on failure.
Prompt Code
# OpenAI JSON mode — guarantees valid JSON object response = client.chat.completions.create( model="gpt-4o-mini", messages=[ {"role": "system", "content": "Extract person info as JSON."}, {"role": "user", "content": bio_text} ], response_format={"type": "json_object"}, temperature=0.0 ) # Output is guaranteed to be valid JSON data = json.loads( response.choices[0].message.content )
✓ API guarantees the response is valid JSON
✓ No markdown wrapping or prose pollution
✓ Can still use json.loads() directly
✗ Doesn't guarantee specific field names/schema
✗ OpenAI-specific feature (not universal)
When to Use This

Use JSON mode when you need valid JSON structure but the exact schema is flexible — for example, when asking the model to summarize information into JSON where the model determines the best structure.

Combine with explicit field instructions in the system prompt to get both valid JSON and predictable field names.

Important: Even in JSON mode, you must describe the fields you want in the prompt. The API enforces JSON syntax, not your schema.
Pydantic + .parse()
from pydantic import BaseModel, Field class PersonInfo(BaseModel): name: str age: int | None occupation: str skills: list[str] location: str | None # Add validation constraints: confidence: float = Field(ge=0.0, le=1.0) response = client.beta.chat.completions.parse( model="gpt-4o-mini", messages=[ {"role": "user", "content": bio_text} ], response_format=PersonInfo, temperature=0.0 ) # Fully typed, validated Python object person = response.choices[0].message.parsed print(person.name, person.skills)
Why This Is Production-Grade
✓ API guarantees JSON conforming to your exact schema
✓ Returns a typed Python object — no parsing needed
✓ Pydantic validators enforce ranges, enums, nullability
✓ IDE autocomplete works on the result object
✗ Requires pydantic dependency
✗ OpenAI-specific .parse() method
Use this for any production code where parsing failures would cause downstream errors. The schema is the contract — the API will enforce it.

JSON Failure Modes & Fixes

The six most common ways structured output generation breaks — and how to fix each.

📦
Format Drift (Markdown Wrapping)
Symptom
Output wrapped in ```json code blocks, causing json.loads() to fail
Example
```json
{"sentiment": "positive"}
```
Fix
Add to prompt: "Return ONLY valid JSON. No markdown, no code fences." Or use json_object mode. Or strip fences with regex.
✍️
Prose Pollution
Symptom
Model explains its answer before/after the JSON: "Here is the analysis: {...} I hope this helps!"
Example
"Based on the review, the sentiment is: {"sentiment": "positive"}"
Fix
Add: "Do not include any explanation, preamble, or commentary. Return only the JSON object."
🔑
Wrong Key Names
Symptom
Model uses "Sentiment" instead of "sentiment", "key_phrase" instead of "key_phrases"
Example
{"Sentiment": "positive", "Phrases": [...]}
Fix
Show exact key names in the prompt schema. Use few-shot examples with the exact schema. Use Pydantic + .parse() to enforce schema.
✂️
Truncated JSON
Symptom
JSON cuts off mid-structure; finish_reason is "length" not "stop"
Example
{"sentiment": "positive", "key_phrases": ["fast", "easy
Fix
Increase max_tokens. Check response.choices[0].finish_reason. For large outputs, paginate or reduce required fields.
📊
Wrong Value Types
Symptom
Confidence returned as "0.95" (string) instead of 0.95 (float); years as "3" instead of 3
Example
{"confidence": "high"} instead of {"confidence": 0.85}
Fix
Specify type explicitly: "confidence: float 0.0–1.0". Use Pydantic for type coercion and validation. Add a few-shot example showing the correct type.
🔄
Inconsistency Across Runs
Symptom
Same input produces different JSON keys or values on different API calls
Example
Run 1: {"sentiment": "mixed"}, Run 2: {"sentiment": "neutral"}
Fix
Set temperature=0.0. Add explicit enum for allowed values. Add a tiebreaker rule in the prompt for ambiguous cases.

Structured Output Best Practices

A checklist for production-quality structured output generation.

Set temperature=0.0
Always use zero temperature for structured outputs. This maximizes consistency across runs of the same input.
Why: Non-zero temperature introduces randomness that causes the same input to produce structurally different outputs.
Specify field types explicitly
Don't assume the model knows your type requirements. Write "confidence: float 0.0–1.0" not just "confidence".
Why: Models default to string for ambiguous fields. Explicit types prevent "0.95" vs 0.95 type errors.
Add "Return ONLY valid JSON"
Explicitly forbid explanations, markdown fences, and preambles in the output format specification.
Why: Without this, models often prefix JSON with explanation text that breaks json.loads().
Validate and retry on failure
Wrap all JSON parsing in try/except. On failure, add an error correction instruction and retry up to 3 times.
Why: Even with perfect prompts, models occasionally produce malformed JSON — defensive parsing prevents silent failures.
Use Pydantic for production code
For any code where parsing failures would cause downstream errors, use Pydantic + .parse() for schema enforcement.
Why: The API guarantees schema conformance — you get a typed Python object with no manual validation code needed.
Test with adversarial inputs
Before deploying, test your structured prompts with empty strings, very long inputs, special characters, and inputs in unexpected languages.
Why: Edge cases that work fine with prose output often break structured output parsers in unexpected ways.
