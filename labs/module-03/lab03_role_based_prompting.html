Important Limitation: Assigning a role does not grant the model real expertise. A "board-certified cardiologist" role uses the same underlying knowledge as a "helpful assistant" role. For high-stakes domains (medical, legal, financial), always add: "This is for informational purposes only. Always consult a licensed professional." Roles shape tone and depth — they don't change what the model actually knows.

Common Role Patterns Reference

A quick reference for the most useful role types and the constraints to add with each.

Senior [Lang] Engineer
Code review, generation, debugging. Produces precise, idiomatic code with explanations calibrated to developer level.
Add: language version, frameworks in use, coding style guide, "no deprecated APIs"
Technical Writer
Documentation, guides, release notes. Produces clear, consistent documentation with proper structure.
Add: "audience: developers", "include code examples", document type (API docs, tutorial, guide)
Data Analyst
Data interpretation, pattern identification, visualization recommendations. Quantitative and precise.
Add: "be quantitative", "highlight outliers", "suggest visualization type"
Legal Reviewer
Contract analysis, risk flagging, compliance checking. Identifies issues without providing formal legal advice.
Add: "flag risks with severity level", "do not give formal legal advice", jurisdiction if relevant
Devil's Advocate
Strategic reviews, pressure-testing plans, identifying blind spots. Deliberately challenges every claim.
Add: "challenge every assumption", "find the weakest point", "do not agree with the user"
Socratic Tutor
Teaching and learning contexts. Guides through questions rather than giving direct answers.
Add: "ask questions, don't give answers directly", "adapt complexity to student's demonstrated level"
